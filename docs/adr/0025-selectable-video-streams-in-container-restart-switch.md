# Selectable video Streams: in-container, capability-defaulted, session-restart switch — no in-band, no ABR

A File may carry more than one **selectable video Stream** (non-cover-art video Streams), sharing one set of audio Streams — e.g. a black-and-white and a colour cut of Spider Noir, or the same picture at two bitrates. The viewer picks one in a new **Video menu**, symmetric to the Audio menu. This is the manual, content-selection sibling of audio Stream selection ([ADR-0022](./0022-audio-stream-selection-in-band-hls-renditions.md)); it is **not** an ABR ladder — the single-video-variant / no-ABR intent of [ADR-0003](./0003-three-tier-playback-with-capability-negotiation.md), [ADR-0004](./0004-hls-for-adaptive-progressive-for-direct-play.md), [ADR-0020](./0020-subtitle-delivery-in-band-hls-out-of-band-track-image-burn-in.md), [ADR-0022](./0022-audio-stream-selection-in-band-hls-renditions.md), [ADR-0024](./0024-direct-stream-video-transcode-audio-fmp4.md) stands.

The video Stream is the selectable unit — **no "Video track" noun is coined** — for the same reason [CONTEXT.md](../../CONTEXT.md) declined "Audio track": with v1 scoped to same-container, embedded Streams are the only source. That noun is reserved for if a second source (a sibling video File) ever arrives.

## Scope — v1

- **Same container only.** The alternate videos live as multiple video Streams in one Matroska/MP4; audio is shared *by construction* (the same audio Streams in the same file, mapped with `-map 0:a:M`). Sibling-file video (video from one File, audio from another) is a documented future extension — it needs a cross-file audio linkage the model does not express today, and would be the "second source" that coins "Video track".
- **Manual selection only.** No automatic, bandwidth-driven switching. Automatic ABR remains refused; if it is ever built, it should be a separate variant ladder over these same Streams.

## Decisions

- **Default video Stream** = the same heuristic that ranks Editions (`SelectEdition`/`better()`): cheapest playable tier first (direct-play/-stream over transcode), then most pixels, then highest bitrate, with the container `is_default` disposition breaking ties among equally-playable Streams. This delivers the "avoid transcoding" half of the multi-bitrate case *for free and without ABR* — a client that cannot decode a 4K HEVC Stream gets a co-packaged 1080p H.264 Stream **direct**, chosen once at negotiation.
- **Remembered video** — a per-User, two-level, meaning-keyed memory mirroring the Remembered audio ([ADR-0023](./0023-per-user-audio-memory-two-level-language-keyed.md)): keyed to the Stream's title tag (falling back to resolution/bitrate), an Episode's pick bubbles up as the Show's, reusing the per-(User, Show) slot ADR-0023 created. It selects *which* content plays; the Capability profile still governs delivery. Resolution order: Title remembered → Show remembered → capability default → `is_default` disposition → first. Unlike audio it has no client-sent hint to outrank.
- **Switching is a full re-negotiation**, not an in-band flip — **there is no user-selectable video rendition in HLS**. A switch re-runs `Negotiate` with the new `videoStreamId` and the *preserved* `audioStreamId`, `burnSubtitleId`, and `startPosition`. It runs normal tier selection and, when it triggers a genuine video encode, **counts against the concurrent-transcode cap and can return `503 SERVER_BUSY`** — the image-subtitle model ([ADR-0020](./0020-subtitle-delivery-in-band-hls-out-of-band-track-image-burn-in.md)), *not* the cap-exempt in-band audio model. The web player reuses its existing escalation-with-possible-busy path.
- **Direct play carries only the default video Stream** (the browser renders its default track; `VideoTrackList` is as unreliable as `audioTracks`). An explicit non-default pick **escalates to HLS remux**, where the server maps the chosen Stream with `-map 0:v:N` — generalizing the demuxed pipeline's hardcoded `-map 0:v:0` ([ADR-0022](./0022-audio-stream-selection-in-band-hls-renditions.md)). Exactly ADR-0022's direct-play-carries-default / escalate-on-explicit-selection posture.
- **Surfaced in-player only.** Selectable video Streams = a File's non-cover-art video Streams; the Video menu auto-appears at ≥2 (like the Audio menu at `audioStreams.length >= 2`); the Decision grows a `videoStreams[]` list parallel to `audioStreams[]`. No per-file configuration and no browse-time badge in v1 (a badge covering multi-video *and* multi-Edition Titles is a future enhancement).

## Why

The whole audio/subtitle switching stack is built on in-band HLS `#EXT-X-MEDIA` renditions the player flips natively with no interruption. **Video has no equivalent**: `#EXT-X-MEDIA:TYPE=VIDEO` is effectively unsupported for user-facing switching in hls.js and native players, and HLS's only real multi-video mechanism — `#EXT-X-STREAM-INF` variants — is the ABR ladder this project has refused end to end. So a video switch is honestly a session restart; modeling it as anything else would be a fiction. Restart is acceptable here (a ~1–2s re-buffer at the same timestamp, like changing quality on YouTube) precisely because video switching is a deliberate, occasional act, not the routine sampling audio switching is.

Modeling the alternates as intra-File video Streams rather than as Editions is what makes audio genuinely *shared*: an Edition bundles its own audio, so B&W and colour as two Editions would duplicate the audio and split watch state. Co-packaged video Streams share one audio set with zero new linkage.

## Consequences

- **The Edition ↔ video-Stream boundary is now explicit.** Two versions that share audio and live in one File are selectable **video Streams** (Video menu, intra-File). Two versions with their own audio in separate Files remain **Editions** (auto-selected by capability/quality, or client `editionId`). The same real-world intent ("offer 1080p and 4K") can be authored either way; packaging decides which concept applies.
- **Public client-API growth**, versioned per [ADR-0003](./0003-three-tier-playback-with-capability-negotiation.md): the negotiation request grows `videoStreamId`; the Decision grows `videoStreams[]`. `pickVideoStream` (`negotiate.go`) becomes `videoStreamId`-aware, and the demuxed builders' `-map 0:v:0` generalizes to `-map 0:v:N`.
- **Watch state widens again** (resume, watched, rating, remembered audio, **remembered video**) — schema and Watch-state API surface change; the glossary is updated. The Show-level slot is the one ADR-0023 already added.
- The denormalized "primary video" columns (`files.video_codec/width/height/bitrate`) still reflect the *first* video Stream; browse-time badges are unaffected in v1 (no video badge shipped).
- **Deferred, deliberately:** automatic ABR and sibling-file (cross-container) shared-audio video. Both are extensions over this same Stream-selection surface, not reversals of it.
