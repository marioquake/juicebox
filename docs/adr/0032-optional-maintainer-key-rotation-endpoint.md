# Optional maintainer-hosted key rotation as a bounded exception to ADR-0001

Juice Box ships **default credentials** for the metadata providers that support redistribution (TMDB, fanart.tv), so external enrichment works out of the box without the operator registering their own API keys. Because the source is open, the plaintext keys live **only in the release pipeline** — injected into official binaries at build time from CI secrets (`-ldflags -X`), never committed to the repo — and may be refreshed at runtime from a **maintainer-hosted rotation endpoint** the operator can disable. This endpoint is a deliberate, bounded exception to [ADR-0001](./0001-fully-self-hosted-no-vendor-dependency.md)'s "no vendor-operated service the operator does not control": it is strictly optional, contacts nothing the operator hasn't consented to, and its unavailability degrades to the exact behavior ADR-0001 already blesses.

## Why this is compatible with ADR-0001, not a breach of it

ADR-0001 forbids a *dependency* on maintainer infrastructure — a relay without which the server cannot function. The rotation endpoint is not that. Enrichment is already **optional and degradeable** under ADR-0001; the endpoint is an enhancement layered on top of an already-optional subsystem, not a new hard edge. Three properties keep it inside the line, and all three are load-bearing — remove any one and this becomes a violation:

- **Never required.** The server functions fully offline. If the endpoint is unreachable, the operator's own key (if set) or the build-time bootstrap key still works; absent both, enrichment degrades to sparse metadata — ADR-0001's blessed fallback. The endpoint can change *which* default key is used; it can never gate whether the server runs.
- **Never new exposure.** The endpoint serves the key encrypted under a symmetric key (`kAppEncKey`) that is itself baked into the same official binary as the bootstrap key. An attacker who can decrypt the endpoint's response already holds the bootstrap key from the binary. So the endpoint adds **rotation, not confidentiality** — its sole job is letting us revoke and replace a leaked default key without shipping a release.
- **Always disable-able.** A config flag turns the endpoint off entirely, and operator-supplied keys (BYOK) bypass it outright. An operator who wants zero maintainer contact has two independent ways to get it, both first-class.

The alternative — no endpoint, rotate only by cutting a release — was rejected not on principle but on incident response: a leaked embedded key would stay live for every not-yet-updated install until they pulled a new Docker tag. Same-day revocation is the entire reason default credentials are safe to ship at all.

## Considered options

- **Embed-only, rotate via release (rejected as sole mechanism).** Purest ADR-0001 fit and retained as the offline fallback layer, but on its own it makes revocation a release-and-wait process — unacceptably slow when a scraped key is being abused against our TMDB/fanart quota.
- **Full metadata proxy (rejected).** The maintainer runs a service that forwards all enrichment requests, so the key never leaves our infrastructure. This is the only option that truly preserves the secret, and it is a direct, unrecoverable ADR-0001 violation: it is exactly the vendor-operated relay ADR-0001 forbids, it makes every install depend on us to function, and it would route every operator's library queries through us — a privacy posture incompatible with a self-hosted server.
- **Plaintext key at a static URL (rejected).** The rotation endpoint's URL is public (it's in the open-source client), so a plaintext response is no better than committing the key — a scraper that finds the URL harvests the key. Serving ciphertext under a build-injected `kAppEncKey` is what makes the public URL safe to expose.
- **Default enrichment silently on (rejected).** Shipping default keys flips enrichment from explicit opt-in to on-by-default, meaning a fresh install would contact TMDB/fanart from the operator's IP with no consent. Rejected in favor of a first-run consent gate (below).

## Consequences

- **Key material never enters the repo.** `bootstrapTMDBKey`, `bootstrapFanartKey`, and `kAppEncKey` are empty package vars in source, populated only by `-ldflags -X` from CI secrets during official builds. A build-from-source install therefore has **no** default keys and uses BYOK — the correct, honest outcome, and it means the open-source repo is genuinely credential-free against scrapers.
- **Credential precedence gains a layer below BYOK.** Resolution order becomes: operator key (env / admin UI) → cached rotation-endpoint key (`data/`, TTL) → build-time bootstrap key → none (sparse metadata). Each layer is a pure fallback for the one above; only the first is guaranteed present.
- **The rotation endpoint is optional infrastructure we now operate**, in tension with ADR-0001's "we provide no infrastructure" posture. This ADR is the explicit, narrowly-scoped license for it: an *optional, disable-able, confidentiality-adding-nothing* rotation channel — and nothing broader. Any future maintainer-hosted service must clear the same three bars (never required, never new exposure, always disable-able) or get its own ADR.
- **Enrichment is now on-by-default behind a first-run consent gate.** Because default keys make enrichment work out of the box, a fresh install must ask before its first external call ("Enable metadata enrichment? Contacts TMDB and fanart.tv"); until the operator consents, no provider — including the rotation endpoint — is contacted. This preserves ADR-0001's "optional, offline-first" guarantee against the new default-on capability. The per-library enrichment policy of [ADR-0027](./0027-per-library-enrichment-policy-sparse-override.md) is unaffected; the gate governs only the global default, which those policies still layer over.
- **Rotation is an operational runbook, not a code path.** Revoking a leaked default key is: revoke on the provider dashboard → re-encrypt the replacement under `kAppEncKey` with the offline tool → publish to the endpoint's store. Rotating `kAppEncKey` itself (a deeper compromise) still requires a release, since it lives in the binary — an intentional asymmetry that keeps routine rotation release-free while bounding the blast radius of the encryption key.
